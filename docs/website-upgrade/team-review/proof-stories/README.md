# Proof Story Workspace

Use this folder to turn Automate4U's real work and real operating knowledge into NDA-safe proof for the website.

The goal is to make proof collection easy for the team:

1. If you only have a few minutes, open `proof-story-quick-capture.md`.
2. If you want the full version, open `proof-story-intake-template.md`.
3. Copy the chosen file into `filled-submissions/`.
4. Rename the copy using a simple project label, for example:
   - `industrial-parts-response-agent.md`
   - `daycare-front-desk-voice.md`
   - `home-services-missed-call-recovery.md`
5. Fill it out with rough notes. Full sentences are not required.
6. Codex can turn the filled version into polished website proof copy.

## Folder Structure

- `proof-story-intake-template.md`: the blank form to copy for each project.
- `proof-story-quick-capture.md`: a short version for fast project notes.
- `proof-story-review-tracker.md`: status board for drafts, approvals, and website placement.
- `proof-opportunity-map.md`: sales pursuit map showing which proof examples should become real offers, pilots, and case studies.
- `founder-experience-proof.md`: how to use founder/operator experience as credibility without confusing it with client case studies.
- `safe-claims-and-metrics-library.md`: approved claim patterns, metric language, and risky language to avoid.
- `asset-redaction-checklist.md`: checklist for screenshots, diagrams, transcripts, and demos before anything is published.
- `filled-submissions/`: where Automate4U should place real project notes.
- `examples/`: pre-filled examples showing the level of detail that is useful.

## Fastest Path

If the team is busy, capture only these six details:

1. Industry and safe public label.
2. What was slow or painful before.
3. Channels and systems involved.
4. What the automation did.
5. What stayed human-controlled.
6. Any safe metric, range, or observable result.

That is enough for a first draft. More detail can be added later.

## What Counts As Useful Detail

Good proof is not just "we built an automation." Good proof explains:

- What was slow before.
- Which channels were involved.
- Which systems or tools mattered.
- What the automation actually did.
- What stayed under human control.
- What changed after the workflow improved.
- Which details must stay private.

## Publication Labels

Use one of these labels before anything goes public:

- **Sanitized case study:** based on one real project, with identifying details removed and approved.
- **Representative workflow pattern:** based on real experience or multiple similar patterns, but not a direct client case study.
- **Example scenario only:** useful for explanation, but should not be presented as completed client work.
- **Internal only:** can help sales calls, but should not be published.
- **Do not use:** too sensitive, too identifying, or not approved.

## Writing Standard

The public version should sound specific and operational, not inflated.

Good:

- "The workflow captures the request, checks approved inventory context, creates a CRM activity, and routes pricing exceptions to staff."
- "Sensitive cases stay with authorized staff, with a summary and audit trail."
- "The useful metric is quote response time, not just chatbot usage."

Avoid:

- "We transformed the business with AI."
- "The AI replaced the team."
- "Guaranteed 10x ROI."
- Any exact client details, proprietary tools, unique process details, or metrics that are not approved.
